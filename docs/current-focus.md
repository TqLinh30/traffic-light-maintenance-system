# Current Focus

- Current task: manually QA the upgraded traffic-light location flows end to end on web and mobile against the resynced live backend on `localhost:8080`, now that location creation/import is no longer capped by the old 10-location license quota, traffic-light location deletion cleans up its QR/point records safely and refreshes the visible web list immediately, the web edit flow can replace an existing pole image from the current Street View framing through backend-downloaded Street View Static images instead of browser fetch, the currently open web traffic-light detail drawer now refetches its point detail immediately after edit saves, the mobile picker has been upgraded to prefer Google Maps plus inline Street View when a `GOOGLE_KEY` is configured, the earlier mobile web bundler blocker from `react-native-image-viewing` has been patched, the web traffic-light location detail surface has been corrected to prefer the fresh `point.locationImageUrl` returned by the traffic-light detail DTO, the local MinIO public endpoint used for signed image URLs has been normalized from `localhost` to `127.0.0.1` in both config and backend runtime signing, the web Street View map sync now uses one Google-style red directional marker instead of the rejected blue camera marker and distance connector, the map-picker address preview is now a persistent lower-left panel that cannot be closed accidentally, and the new traffic-light `Add Location` and `Edit Location` modals no longer crash on mount from a recursive Formik update loop.
- Why: the project now treats each managed location as a traffic-light pole, so both web and mobile operators need create or edit flows that capture coordinates, image, and lifecycle inputs consistently, plus detail surfaces that foreground pole information instead of generic location or asset-first data. The live backend is synced to the latest code and now returns `locationImageUrl` plus lifecycle metadata on the traffic-light detail DTOs, but older sample locations like `test04` still have `NULL` lifecycle and image values until they are repaired through the new edit flow or recreated. The web traffic-light detail tab had still been rendering `location.image.url` from the broader location payload, which could be missing or stale even when the dedicated traffic-light detail fetch already contained a fresh signed image URL. After that UI fix, local browser QA still showed broken-image icons for rows such as `test07`; direct database and MinIO inspection confirmed the image file existed, but presigned URLs using `http://localhost:9000` were being reset by the local Windows or Docker network path while the same object loaded correctly when signed against `http://127.0.0.1:9000`. A later live API check showed the restarted backend could still inherit `PUBLIC_MINIO_ENDPOINT=http://localhost:9000`; `MinioService` now normalizes that stale local public endpoint to `127.0.0.1` before creating the signing client, and the restarted backend has been verified to return a `127.0.0.1:9000` image URL that loads as `200 image/jpeg`. The latest Street View QA rejected the dual-marker design because the blue camera dot and connector made the selected point look separate from the live Google Street View position. The corrected web flow now treats the active Street View camera position as the selected map position during add and edit, updates the map popup coordinates from the same state, and renders a single red marker with a heading cone similar to Google Maps. The fallback image path still uses the current Street View position, heading, pitch, and zoom, while loop guards prevent the panorama from reloading itself when those coordinates are written back into Formik. A follow-up local render harness also exposed a regression in the dedicated web traffic-light form: `StreetViewPreview` depended directly on callback props that called `formik.setFieldValue(...)`, so the mount-time `onCaptureChange(null)` path could recursively retrigger the effect and blank the entire modal with `Maximum update depth exceeded`. The preview now keeps the latest callbacks in refs so the form can mount safely while still reacting to real Street View movement. The latest web mutation follow-up fixed a stale-table issue where delete and edit only updated the flat `locations` state while the visible table rendered from `locationsHierarchy`, and it changed edit image handling so a generated Street View image can replace an existing pole image when no manual upload is provided. A browser QA pass then exposed that converting Google Street View Static images to base64 through frontend `fetch()` can fail with `Failed to fetch`; the form now sends a restricted Street View Static URL and the backend downloads the image directly before saving it. The mobile picker originally used `Leaflet + OpenStreetMap + Nominatim` inside a `WebView`; it now prefers Google Maps JS and inline Street View to get much closer to the accepted web workflow while keeping the earlier keyless OpenStreetMap fallback path when no `GOOGLE_KEY` is present in the mobile runtime.
